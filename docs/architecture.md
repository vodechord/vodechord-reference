# Architecture

This document describes the technical stack for the Listening Room, how it's structured, how it's built, and how it's deployed. Decisions here are deliberately conservative — the site should be cheap to run, boring to maintain, and survivable without the author's active attention.

---

## Stack Decisions

### Frontend: vanilla HTML/CSS/JS, no framework

No React, Vue, Svelte, Next, Astro, or anything else with a build step more complicated than "concatenate and minify."

Why:

- The site is mostly static content. Framework machinery is overkill.
- Vanilla JS works indefinitely; frameworks rot on a 2–3 year cycle.
- Web Audio API is a native browser API. Wrapping it in a framework adds indirection without benefit.
- BlueHost shared hosting serves static files well and doesn't run Node.js reliably. A static build is the natural fit.
- Faster to ship — no dependency graph to maintain.

Concession: a small bundler (esbuild) or even concatenation-by-shell-script can be used if the JS grows past a dozen files. No virtual DOM, no JSX, no component framework.

### Styling: hand-written CSS, no Tailwind

For the same reasons. Modern CSS (custom properties, grid, flex, container queries) handles everything a reference site needs. A single stylesheet with thoughtfully-named classes is easier to read six months from now than a Tailwind-flavored DSL.

### Audio: Web Audio API for real-time exhibits, pre-rendered WAV for the rest

- **Real-time exhibits** use `AudioContext`, `OscillatorNode`, `BiquadFilterNode`, `AudioWorkletNode` (for custom DSP), and `AudioBufferSourceNode`. Implemented as plain JS modules in `/src/exhibits/`.
- **Pre-rendered examples** are WAV files generated by Python scripts, committed to `/audio/generated/` or regenerated on CI.
- **Curated references** are never hosted; the site embeds or links to original sources.

### Audio generation scripts: Python

Python for anything that runs server-side or at build time. Reasons:

- Excellent DSP library ecosystem: `numpy`, `scipy`, `librosa`, `soundfile`, `praat-parselmouth`, `pyworld`, `crepe`.
- The same scripts are useful in later Vodechord prototypes.
- Deterministic output (given a random seed) enables reproducible audio — important for CI.

Generation scripts live in `/scripts/`. Each script takes command-line args for parameters, emits one or more WAV files to `/audio/generated/`, and uses fixed random seeds by default.

### No database. No backend. No server-side rendering.

Pure static site. If it ever needs a backend — for analytics, user accounts, rights requests — we revisit. For now, deliberately zero server-side state.

## Build Pipeline

The "build" is deliberately minimal. Three steps:

1. **Render audio.** Run each generation script in `/scripts/`, producing WAV files in `/audio/generated/`.
2. **Concatenate & minify.** A build script concatenates HTML fragments from `/src/pages/` into full pages, inlines CSS if small, and minifies JS with `esbuild`. Output goes to `/build/`.
3. **Copy assets.** `/audio/generated/` and any static assets are copied into `/build/`.

A single `scripts/build.sh` orchestrates all three. The output in `/build/` is what ships to BlueHost. Everything needed to recreate `/build/` from source lives in the repo.

### Audio rendering on CI vs locally

Audio generation is slow-ish but deterministic. Two options:

- **Commit generated WAVs to the repo.** Pros: `/build/` step is trivial. Cons: repo bloat.
- **Regenerate on every CI build.** Pros: small repo. Cons: slower CI, generation scripts must be robust and reproducible.

**Decision for v1: commit generated WAVs.** Audio totals are expected to stay under ~50 MB for the full site, which is fine for git. If that grows, we revisit. Regeneration is triggered manually by the developer when a generation script changes, via `make render-audio` or equivalent.

## Deployment to BlueHost

BlueHost serves static files via standard HTTP/HTTPS. The site deploys as follows:

1. **GitHub Actions CI** runs on push to `main`.
2. On success, CI runs the build and produces `/build/`.
3. CI uses **rsync over SSH** (or SFTP as fallback) to upload `/build/` to the BlueHost server's `public_html/listening-room/` directory.
4. BlueHost serves the result at `https://[your-domain]/listening-room/`.

### Secrets

GitHub Actions needs:

- `BLUEHOST_HOST` — the server hostname
- `BLUEHOST_USER` — the SSH username
- `BLUEHOST_SSH_KEY` — the private key (added to Actions secrets)
- `BLUEHOST_PATH` — the target directory on the server

These are stored as GitHub Actions secrets. The public repo never contains them. The user (Jerome) will need to set these up in the repo's Settings → Secrets page before the first deploy.

### No downtime / rollback

The upload is not atomic — rsync overwrites in place. If a deploy corrupts the site, the fix is another push. This is acceptable for a small static reference site; a real product would want atomic swaps via symlinks.

A simple safeguard: before rsync, CI takes a `tar.gz` snapshot of the current remote `listening-room/` to `backup/listening-room-<date>.tar.gz` so recovery is one SSH command away.

## CI Pipeline

GitHub Actions workflow: `.github/workflows/ci.yml`.

Triggers:
- On push to any branch: lint + test.
- On push to `main`: lint + test + build + deploy.

Jobs:

1. **Lint** — shellcheck on shell scripts, flake8/ruff on Python, eslint on JS, stylelint on CSS. Fast.
2. **Audio reproducibility test** — regenerate a handful of WAVs and check they match committed copies (via hash). Ensures generation scripts stay reproducible.
3. **JS unit tests** (later) — any DSP-logic helpers in `src/js/audio.js` should be testable. The Web Audio exhibits themselves are not unit-testable in CI; manual listening QA covers them.
4. **Link check** — scan markdown and HTML for broken external links. Important because the site is link-heavy.
5. **Build** — run `scripts/build.sh`, verify output.
6. **Deploy** (main only) — rsync `/build/` to BlueHost.

Deploy failure does not fail the CI build (to avoid "red build, unclear why"), but it posts a clear summary.

## Performance

- Total page weight: target under 2 MB initial load per section, including audio.
- Audio files: mono 22050 Hz Opus or MP3 for curated; WAV for generated (size acceptable).
- Lazy-load audio on demand — don't fetch a section's WAVs until the listener navigates there.
- Web Audio exhibits spin up their `AudioContext` only on user gesture (required by browsers anyway).

---

## Accessibility

- All audio has text descriptions of what the listener should hear.
- Real-time exhibits have keyboard controls in addition to slider drag.
- Color is not the only channel for state (playing/paused, selected/unselected).
- Semantic HTML throughout.
- No autoplay.

---

## Browser Support

- **Target:** evergreen Chrome, Safari, Firefox, Edge. Latest two versions.
- **Web Audio API support is universal** on those browsers; `AudioWorkletNode` (used by some grit exhibits) is supported everywhere but IE (not a target).
- **Mobile:** functional but not optimized in v1. Touch sliders work; layout needs a pass. Explicit "best on desktop with headphones" note on mobile.

---

## Future Considerations

Intentionally deferred:

- **Analytics.** None in v1. Plausible or similar if useful later.
- **RSS / changelog feed.** A good-to-have once the site has regular updates.
- **Multi-language.** Not planned.
- **Downloadable sample packs.** Could be a natural extension once generated audio is robust.
- **Forum or comments.** No.

## Setup Checklist

For the author / first-time contributor:

1. Clone the repo locally.
2. Install Python 3.10+ and Node 18+.
3. `pip install -r scripts/requirements.txt`
4. `npm install` (if a `package.json` exists for esbuild — to be added later).
5. Run `scripts/render-audio.sh` to generate audio (first time; slow).
6. Run `scripts/dev.sh` to serve `/build/` locally with live reload.
7. Edit pages in `/src/pages/`, add or edit exhibits in `/src/exhibits/`, regenerate audio if a generation script changes.

When ready to deploy:

1. Configure GitHub Actions secrets (one-time).
2. Push to `main`.
3. Verify deploy at `https://[your-domain]/listening-room/`.
